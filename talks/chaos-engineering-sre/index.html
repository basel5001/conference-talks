Why Chaos Engineering?


"Chaos Engineering is the discipline of experimenting on a system in order to build confidence in the system's capability to withstand turbulent conditions in production."

-- Principles of Chaos Engineering

Netflix pioneered this with Chaos Monkey. The idea is simple: break things on purpose so you can fix them before they break on their own.

5 Experiment Types

A systematic approach to breaking things

The 5 Experiment Types


  1. Pod Failure -- Kill pods, verify self-healing
  2. Network Chaos -- Latency, partition, packet loss
  3. Resource Stress -- CPU, memory, disk exhaustion
  4. DNS Failure -- Break name resolution
  5. Zone Outage -- Simulate AZ failure

Experiment 1: Pod Failure



# Hypothesis: If we kill 50% of API pods,
# the service remains available with < 500ms latency

apiVersion: chaos.experiment/v1
kind: PodFailure
metadata:
  name: api-pod-kill
spec:
  target:
    namespace: production
    labelSelector:
      app: api-gateway
    mode: percentage
    value: 50
  duration: 5m
  steadyState:
    - probe: http
      url: https://api.example.com/health
      expectedStatus: 200
      maxLatency: 500ms
      interval: 10s
  rollback:
    automatic: true

Orchestrator Core



class ChaosOrchestrator:
    def __init__(self):
        self.experiments = ExperimentRegistry()
        self.safety = SafetyController()
        self.analyzer = AIAnalyzer()

    async def run_experiment(
        self, experiment: Experiment
    ) -> ExperimentResult:

        # Pre-flight safety checks
        if not await self.safety.pre_check(experiment):
            return ExperimentResult(
                status="blocked",
                reason="Safety check failed",
            )

        # Record steady state
        baseline = await self.measure_steady_state(
            experiment.steady_state_probes
        )

        # Inject fault
        try:
            await self.inject_fault(experiment)
            await asyncio.sleep(experiment.duration)
        finally:
            # Always rollback
            await self.rollback(experiment)

        # Measure impact
        result = await self.measure_impact(baseline)
        return result

Safety Controls



class SafetyController:
    """Circuit breaker for chaos experiments."""

    MAX_BLAST_RADIUS = 0.5  # Never affect > 50%
    MIN_HEALTHY_REPLICAS = 1
    BUSINESS_HOURS_ONLY = True

    async def pre_check(self, exp: Experiment) -> bool:
        checks = [
            self._check_blast_radius(exp),
            self._check_healthy_replicas(exp),
            self._check_business_hours(),
            self._check_no_active_incidents(),
            self._check_approval_granted(exp),
        ]

        results = await asyncio.gather(*checks)

        if not all(results):
            failed = [c for c, r in zip(
                ["blast_radius", "replicas", "hours",
                 "incidents", "approval"], results
            ) if not r]
            logger.warning(f"Safety checks failed: {failed}")
            return False
        return True
Never run chaos without safety controls. The goal is controlled failure, not actual outages.

AI-Powered Analysis

Making sense of chaos results

AI Post-Experiment Analysis



class AIAnalyzer:
    def __init__(self):
        self.bedrock = boto3.client("bedrock-runtime")

    async def analyze_results(
        self, experiment: Experiment,
        baseline: Metrics, observed: Metrics
    ) -> Analysis:

        prompt = f"""Analyze this chaos experiment:

        Experiment: {experiment.type}
        Target: {experiment.target}
        Duration: {experiment.duration}

        Baseline metrics:
        {json.dumps(baseline.to_dict())}

        Observed metrics during chaos:
        {json.dumps(observed.to_dict())}

        Provide: resilience score (1-10), failure modes found,
        recommendations, and whether the hypothesis held.
        """

        return Analysis.from_response(
            self._invoke_model(prompt)
        )

Running Your First Experiment

A practical guide to getting started

Start Here


  1. Pick a non-critical service -- Start in staging
  2. Define the hypothesis -- What should happen?
  3. Set up monitoring -- You need observability first
  4. Start with pod failure -- Simplest experiment
  5. Run during business hours -- People are around to fix things
  6. Review and improve -- Each experiment teaches something



# Your very first chaos experiment

Key Takeaways


  1. Start small, fail safely -- Pod kills before zone failures
  2. Hypothesis-driven -- Always define expected behavior
  3. Safety first -- Circuit breakers are non-negotiable
  4. AI accelerates learning -- Pattern recognition at scale
  5. Chaos is a culture, not a tool
