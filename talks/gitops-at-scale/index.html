Welcome everyone. Today I'm going to share how we manage 22 repositories across multiple teams using a single TypeScript configuration file and GitOps principles.

The Problem


22 Repositories

  • Inconsistent branch protection
  • Manual security settings
  • Drift between repos
  • No audit trail

Multiple Teams

  • Different naming conventions
  • Inconsistent CI/CD
  • Security gaps
  • "It works on my repo"
Before GitOps, every repo was a snowflake. Settings applied manually through the UI. No way to know if a repo was compliant without checking each one.

What is GitOps?

Git as the single source of truth for everything

GitOps Principles


  1. Declarative - Describe the desired state
  2. Versioned - Git tracks all changes
  3. Automated - Changes applied automatically
  4. Self-healing - Drift is detected and corrected
These four principles guide everything we build. If it's not in Git, it doesn't exist.

ArgoCD - Continuous Deployment



apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: gitops-cluster
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/basel5001/gitops-cluster
    targetRevision: main
    path: manifests
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
ArgoCD watches Git and automatically syncs the cluster state. Self-heal means if someone makes a manual change, it gets reverted.

Kyverno - Policy Engine



apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-labels
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-team-label
      match:
        any:
          - resources:
              kinds:
                - Deployment
                - Service
      validate:
        message: "Label 'team' is required"
        pattern:
          metadata:
            labels:
              team: "?*"
Kyverno enforces policies as Kubernetes resources. No separate policy language to learn.

Progressive Delivery

Argo Rollouts for safe deployments

Canary Deployments with Argo Rollouts



apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: api-service
spec:
  strategy:
    canary:
      steps:
        - setWeight: 10
        - pause: { duration: 5m }
        - setWeight: 30
        - pause: { duration: 5m }
        - setWeight: 60
        - pause: { duration: 5m }
        - setWeight: 100
      canaryService: api-canary
      stableService: api-stable
      trafficRouting:
        istio:
          virtualService:
            name: api-vsvc
Traffic shifts gradually. If metrics degrade, automatic rollback kicks in.

Automated Analysis



apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: success-rate
spec:
  metrics:
    - name: success-rate
      interval: 60s
      successCondition: result[0] >= 0.95
      provider:
        prometheus:
          address: http://prometheus:9090
          query: |
            sum(rate(http_requests_total{
              status=~"2.*",
              service="{{args.service}}"
            }[5m])) /
            sum(rate(http_requests_total{
              service="{{args.service}}"
            }[5m]))

Automatic rollback if success rate drops below 95%

Lessons Learned

Lesson 1: Start Small


"Don't try to GitOps everything on day one."

  • Start with repository settings (low risk)
  • Add branch protection next
  • Then CI/CD pipelines
  • Finally, infrastructure (Crossplane)

Lesson 2: Type Safety Saves You



// TypeScript catches mistakes at compile time
interface RepoConfig {
  name: string;
  description: string;
  visibility: "public" | "private";  // Not just string
  branchProtection: BranchRules;
  security: SecurityConfig;
}

// Typo? TypeScript catches it.
// Wrong type? TypeScript catches it.
// Missing required field? TypeScript catches it.

YAML doesn't give you this. TypeScript does.

Lesson 3: Escape Hatches Matter


  • Not everything fits the standard template
  • Allow per-repo overrides
  • Document exceptions in the config file
  • Review overrides in PRs



{
  name: "legacy-monolith",
  // Override: needs different branch protection
  // because of legacy CI system - tracked in JIRA-1234
  branchProtection: {
    main: { ...standardProtection, requiredReviews: 2 },
    develop: { requiredReviews: 0 },  // legacy flow
  },
}

Key Takeaways


  1. Git is your source of truth - for everything
  2. TypeScript > YAML - for config management
  3. Automate compliance - don't rely on humans
  4. Progressive delivery - ship safely at scale
  5. Start small, iterate - don't boil the ocean
